# /learn - Codebase Pattern Analyzer & Rule Generator

The most powerful skill in CodePilot. Analyzes YOUR codebase and generates custom rules so Claude follows YOUR conventions, not generic best practices.

## What This Does

1. Scans your actual code for patterns and conventions
2. Identifies naming, structure, and architectural patterns
3. Generates `.claude/rules/` files tailored to your project
4. Claude then enforces YOUR patterns in all future code generation

## Phase 1: Deep Codebase Scan

```bash
echo "=== CODEBASE PATTERN ANALYSIS ==="

# File organization
echo "--- File Organization ---"
find . -type f \( -name "*.ts" -o -name "*.tsx" -o -name "*.js" -o -name "*.jsx" -o -name "*.py" -o -name "*.go" \) | grep -v node_modules | grep -v .next | grep -v __pycache__ | grep -v dist | head -50

# Naming conventions
echo ""
echo "--- File Naming ---"
find . -type f -name "*.ts" -o -name "*.tsx" | grep -v node_modules | grep -v .next | xargs -I{} basename {} | sort -u | head -30

# Component patterns
echo ""
echo "--- Component Patterns ---"
grep -rn 'export default function\|export function\|export const.*=.*=>' --include="*.tsx" . 2>/dev/null | grep -v node_modules | head -20

# Import patterns
echo ""
echo "--- Import Patterns ---"
grep -rn '^import ' --include="*.ts" --include="*.tsx" . 2>/dev/null | grep -v node_modules | grep -v '.next' | head -30

# Export patterns
echo ""
echo "--- Export Patterns ---"
grep -rn '^export ' --include="*.ts" --include="*.tsx" . 2>/dev/null | grep -v node_modules | grep -v '.next' | head -30

# Error handling
echo ""
echo "--- Error Handling ---"
grep -rn 'try\s*{' --include="*.ts" --include="*.tsx" . 2>/dev/null | grep -v node_modules | head -10
grep -rn 'catch\s*(' --include="*.ts" --include="*.tsx" . 2>/dev/null | grep -v node_modules | head -10

# State management
echo ""
echo "--- State Management ---"
grep -rn 'useState\|useReducer\|useContext\|zustand\|create(\|atom(' --include="*.ts" --include="*.tsx" . 2>/dev/null | grep -v node_modules | head -10

# API patterns
echo ""
echo "--- API Patterns ---"
grep -rn 'fetch(\|axios\.\|useSWR\|useQuery\|trpc\.' --include="*.ts" --include="*.tsx" . 2>/dev/null | grep -v node_modules | head -10

# Testing patterns
echo ""
echo "--- Test Patterns ---"
grep -rn 'describe(\|it(\|test(\|expect(' --include="*.test.*" --include="*.spec.*" . 2>/dev/null | grep -v node_modules | head -10
```

## Phase 2: Identify Patterns

Analyze the scan results to identify:

### A. Naming Conventions
- File names: kebab-case, camelCase, PascalCase, snake_case?
- Component names: PascalCase?
- Function names: camelCase, snake_case?
- Variable names: camelCase, UPPER_CASE for constants?
- CSS classes: Tailwind, BEM, CSS modules?
- Route naming: plural nouns, kebab-case?
- Test file naming: `.test.ts`, `.spec.ts`, `__tests__/`?

### B. Architecture Patterns
- Folder structure: feature-based, layer-based, hybrid?
- Components: function components, arrow functions?
- Exports: default export, named export, barrel files?
- State: local state, global store, server state?
- Data fetching: Server Components, SWR, React Query, tRPC?
- Forms: controlled, uncontrolled, server actions?
- Error handling: try/catch, error boundaries, Result type?

### C. Code Style
- Quotes: single, double?
- Semicolons: yes, no?
- Indentation: tabs, 2 spaces, 4 spaces?
- Trailing commas: yes, no?
- Import ordering: framework → external → internal → relative?
- Type annotations: explicit return types? Interface vs Type?

### D. Project-Specific Patterns
- Custom hooks naming: `useX`?
- Service layer patterns: class, function, object?
- Validation: Zod, Yup, custom?
- Authentication pattern: middleware, HOC, hook?
- Database access pattern: repository, direct ORM, service layer?
- Logging pattern: console, structured logger, custom?

## Phase 3: Generate Custom Rules

Based on the analysis, generate project-specific rules in `.claude/rules/`:

### Example Generated Rule: `project-conventions.md`

```markdown
---
paths:
  - "**/*.ts"
  - "**/*.tsx"
---

# Project Conventions (Auto-generated by /learn)

## Naming
- Files: kebab-case (e.g., `user-profile.tsx`, `auth-service.ts`)
- Components: PascalCase function components (e.g., `export function UserProfile()`)
- Hooks: `useX` prefix (e.g., `useAuth`, `useProducts`)
- Types/Interfaces: PascalCase with `interface` keyword (not `type` for objects)
- Constants: UPPER_SNAKE_CASE (e.g., `MAX_RETRIES`)
- API routes: plural nouns, kebab-case (e.g., `/api/user-profiles`)

## Structure
- Components use named exports (not default exports)
- Barrel files (index.ts) for component directories
- Co-located tests: `component.test.tsx` next to `component.tsx`
- Services in `src/server/services/`
- Shared types in `src/types/`

## Patterns
- Data fetching: Server Components with async/await (no useEffect for data)
- Forms: Server Actions with `useActionState`
- Validation: Zod schemas co-located with forms
- State: Zustand for global state, useState for local
- Styling: Tailwind utility classes, cn() helper for conditional classes

## Error Handling
- API routes: try/catch with structured error responses
- Client: Error boundaries at route level
- Forms: Return { error: string } from server actions

## Imports Order
1. React/Next.js
2. External packages
3. Internal aliases (@/lib, @/components)
4. Relative imports
5. Type imports last
```

### Example Generated Rule: `database-conventions.md`

```markdown
---
paths:
  - "src/server/**"
  - "prisma/**"
---

# Database Conventions (Auto-generated by /learn)

## Prisma
- Use `prisma.$transaction()` for multi-step operations
- Always `select` or `include` specific fields (no `findMany()` without selection)
- Soft delete pattern: `deletedAt` field instead of actual deletion
- Pagination: cursor-based for lists, offset for admin panels

## Queries
- N+1 prevention: Always use `include` for related data in list queries
- Index hints: Add `@@index` for frequently filtered fields
- Enum for status fields (not string)
```

## Phase 4: Generate Rules

Read the top 10-20 most representative files in the project to extract real patterns. Then generate rule files that capture:

1. **`project-conventions.md`** — Naming, structure, style (scoped to all files)
2. **`component-patterns.md`** — UI component conventions (scoped to `.tsx` files)
3. **`api-patterns.md`** — API/backend conventions (scoped to API directories)
4. **`testing-patterns.md`** — Testing conventions (scoped to test files)

Write them to `.claude/rules/learned/` to keep them separate from manually written rules.

## Phase 5: Validate & Report

```bash
echo "=== GENERATED RULES ==="
find .claude/rules/learned/ -name "*.md" -exec echo "File: {}" \; -exec wc -l {} \; 2>/dev/null

echo ""
echo "Rules will take effect immediately for all Claude Code interactions."
```

RULE: Only generate rules from patterns with HIGH or MEDIUM confidence. Don't enforce inconsistent patterns.
RULE: Never overwrite manually-written rules in `.claude/rules/`. Only write to `.claude/rules/learned/`.
RULE: Read at least 10 actual source files before generating rules. Don't guess from file names alone.
RULE: Include the confidence level and sample count in generated rule comments so developers can verify.
